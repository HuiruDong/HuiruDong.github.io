重学前端笔记（8）

前端 浏览器
学习笔记
发布日期:   2020-11-11

到了新的篇章《浏览器的实现原理和 API》，当时看目录的时候第一个想到的问题就是能不能系统的说一下“URL 是怎么显示在屏幕上的”，这个真的困扰了我很久。没想到第一篇的第一个问题就是这个问题，开心~

敲入网址后究竟发生了什么？

  1. 浏览器首先使用 HTTP 协议或者 HTTPS 协议，向服务端请求页面
  2. 解析请求回来的 HTML 代码，构建成 DOM 树
  3. 计算 DOM 树上的 CSS 属性
  4. 根据 CSS 属性对元素逐个选人啊，得到内存中的位图
  5. 对位图进行合成，这会极大地增加后续绘制的速度
  6. 合成后，绘制到界面上

常规来说我们看到这个流程的时候感觉是一步一步的做，实际上这个过程是流水线式的。从请求回来，就产生了数据流，后续步骤都是在流式处理数据，不用非要上一步完成之后再做下一步，这就是为什么有时候页面会逐步出现。

下面我们具体的说说每个部分

HTTP 协议

HTTP 协议是基于 TCP 协议出现的，TCP 协议是一条双向的通讯通道，HTTP 在 TCP 的基础上，规定了 Request-Response 的模式。这个模式决定了通讯必定是由浏览器端首先发起的。HTTP 是纯粹的文本协议，它是规定了使用 TCP 协议来传输文本格式的一个应用层协议。

Method

  • GET
  • POST
  • HEAD：只返回请求头，所以一般由 JS 发起
  • PUT
  • DELETE
  • CONNECT：多用于 HTTPS 和 WebScoket
  • OPTIONS
  • TRACE

Status Code

  • 1xx：临时回应，请继续
  • 2xx
    • 200
  • 3xx：请求的目标有变化
    • 301：永久性跳转，换而言之就是客户端你别来了
    • 302：临时性跳转
    • 304：缓存没更新，继续用你的缓存
  • 4xx
    • 403：无权限
    • 404
  • 5xx
    • 500
    • 503：暂时性错误，要不等会再试试？

Body

一些常见的格式

  • application/json
  • appliction/x-www-form-urlencoded：表单提交
  • multipart/form-data：文件上传
  • text/xml

HTTPS

HTTPS 是使用加密通道来传输 HTTP 的内容。但是 HTTPS 首先与服务端建立一条 TLS 加密通道。TLS 构建于 TCP 协议之上，它实际上是对传输的内容做一次加密，所以从传输内容上看，HTTPS 跟 HTTP 没有任何区别。

HTTPS 有两个作用

  • 确定请求的目标服务端身份
  • 保证传输的数据不会被网络中间节点窃听或篡改

HTTP 2

HTTP 2 是 HTTP 1.1 的升级版本，最大的改进有两点：

  • 支持服务端推送
    • 服务端推送能够在客户端发送第一个请求到服务端的时候，提前把一部分内容推送给客户端，放在缓存里面，这样可以避免客服端请求顺序带来的并行度不高问题，导致性能上问题
  • 支持 TCP 连接复用
    • 使用同一个 TCP 连接来传输多个 HTTP 请求，避免了 TCP 连接建立时的三次握手的开销和初建 TCP 连接时传输窗口小的问题
